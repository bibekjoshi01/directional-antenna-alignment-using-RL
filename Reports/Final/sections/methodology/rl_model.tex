\subsection{Reinforcement Learning Model}

The core of the autonomous antenna alignment system is a lightweight reinforcement
learning (RL) agent that adapts the receiver antenna orientation to maximize received
signal strength. Unlike conventional control methods, RL enables the system to learn a
useful alignment policy from interaction data without relying on an explicit channel
model.


\subsubsection{RL Framework Overview}

RL is a learning paradigm in which an agent learns to make sequential decisions by
interacting with an environment and maximizing cumulative reward. Unlike supervised
learning, where correct input-output pairs are provided, RL relies on scalar reward
feedback indicating whether an action improved or degraded performance. This framework
is particularly suitable for control problems in which explicit models are unavailable
or impractical to obtain~\cite{ref19}.

Model-free reinforcement learning methods, such as Q-learning, estimate the expected
long-term return associated with state-action pairs without requiring an explicit model
of the environment. Such methods are attractive here because the final controller must
execute on an embedded platform with limited memory and deterministic runtime
requirements. In this project, learning is performed offline and only the trained
Q-table is deployed to the hardware.

\subsubsection{Q-Learning}

Q-learning was selected because it provides a good balance between learning capability,
implementation simplicity, and embedded feasibility for the present problem. The antenna
alignment task uses a compact discrete state-action space, and the controller must run
with predictable memory and computation on the ESP32-S3. Under these constraints,
tabular Q-learning can be trained offline and then deployed as a fixed lookup table with
very low runtime overhead.

SARSA was considered less suitable because it is an on-policy method, meaning that its
updates depend on the same exploratory policy used during training. This often produces
more conservative behavior, which can be useful in safety-critical domains, but in the
present work the objective was to learn a strong greedy deployment policy from offline
simulation and then execute it deterministically on hardware. Q-learning is more aligned
with this objective because it directly estimates the value of the greedy target policy.

Deep Q-Networks (DQN) were also not adopted because their advantages are more relevant
to large or continuous state spaces. In this project, the state space is intentionally
kept small through discretization, so a neural network would add training complexity,
hyperparameter sensitivity, and higher deployment cost without a proportional practical
benefit. DQN would also complicate on-device inference and reduce transparency compared
with a tabular policy that can be inspected directly.


The problem is formulated as a Markov Decision Process (MDP) comprising:
\begin{itemize}
    \item \textbf{States ($s_t$):} Each state represents the current azimuth index,
    tilt index, and recent RSSI trend class.
    \item \textbf{Actions ($a_t$):} Discrete antenna adjustments consisting of pan
    left, pan right, tilt up, tilt down, and hold.
 \item \textbf{Reward ($r_t$):} Scalar feedback representing the change in
    averaged RSSI after taking an action, together with a small penalty on movement.
    \item \textbf{Policy:} During training, an $\epsilon$-greedy policy balances
    exploration and exploitation. During deployment, greedy action selection is used.
\end{itemize}

\subsubsection{Exploration--Exploitation Trade-off}

The RL agent is trained using an $\epsilon$-greedy strategy so that it can first explore
the search space and later exploit the best actions discovered during learning. A high
initial exploration rate is necessary because the agent must experience both favorable
and unfavorable antenna orientations in order to learn a meaningful Q-table rather than
overfit to a few starting positions.

As training progresses, $\epsilon$ is gradually reduced so that the updates increasingly
reflect exploitation of higher-value actions. This decay is justified because once the
agent has sampled the practical state space sufficiently, continuing strong exploration
would only slow convergence of the Q-values and inject unnecessary randomness into the
learned policy. During deployment, exploration is disabled entirely so that the embedded
controller behaves repeatably and does not intentionally take suboptimal actions on
hardware.

\subsubsection{Training Convergence}

Training convergence in this work refers to stabilization of the learned Q-table and the
associated improvement trends over repeated episodes. In practice, convergence was judged
from the reward-per-episode and best-RSSI curves: once these curves stopped showing
meaningful long-term improvement and the policy repeatedly reached high-RSSI regions from
diverse initial states, the Q-table was treated as sufficiently converged for export.

This practical notion of convergence is appropriate here because the objective is not a
formal proof of optimality in an abstract MDP, but a stable policy that can be deployed
reliably on embedded hardware. The final number of training episodes and the exploration
decay schedule were therefore chosen to produce repeatable convergence behavior without
excessive offline training cost.

\subsubsection{Q-Learning Algorithm Pseudocode}

The following code-style pseudocode summarizes the adopted Q-learning training and
deployment procedure.

\begin{minted}[linenos, fontsize=\small]{c}
# Initialize Q-table
Initialize Q[s, a] for all states s and actions a
Set alpha (learning rate), gamma (discount factor), 
epsilon (exploration rate)

for each episode:
    reset environment to a valid random initial orientation
    
    measure averaged RSSI
    encode current state s

    for each step until terminal condition or max_steps:
        
        # Action selection (epsilon-greedy)
        with probability epsilon:
            choose a random action a
        else:
            choose a = argmax_a Q[s, a]

        # Apply action
        apply action a
        
        measure next averaged RSSI
        compute reward r from RSSI change and movement penalty
        
        encode next state s_next

        # Q-learning update
        Q[s, a] = Q[s, a] + alpha * (
            r + gamma * max_a' Q[s_next, a'] - Q[s, a]
        )

        s = s_next

    # Exploration decay
    decay epsilon

# Export trained model
export converged Q-table to ESP32-S3 firmware

# Deployment (inference only)
during deployment:
    choose a = argmax_a Q[s, a]
    do not perform online learning
\end{minted}

\subsubsection{State Space Design}
The state space represents the information available to the RL agent at each decision
step. To maintain a compact representation suitable for the ESP32-S3, the deployed
controller uses the practical motion limits of the hardware rather than a full
$360^{\circ}$ scan:
\begin{itemize}
    \item Azimuth angle is discretized into $N_\theta = 19$ positions over
    $0^{\circ}$ to $180^{\circ}$ at $10^{\circ}$ resolution
    \item Tilt angle is discretized into $N_\phi = 11$ positions over
    $25^{\circ}$ to $75^{\circ}$ at $5^{\circ}$ resolution
    \item RSSI trend is encoded into $N_r = 3$ classes: increasing, stable, and
    decreasing
\end{itemize}
The final state vector is a combination of azimuth index, tilt index, and RSSI trend
index:
\begin{equation}
s_t = \left(i_{\theta},\, i_{\phi},\, i_r \right)
\end{equation}
The number of distinct states is therefore:
\begin{equation}
\left|S\right| = N_\theta \times N_\phi \times N_r = 19 \times 11 \times 3 = 627
\end{equation}

\subsubsection{Q-Table Size and Memory Footprint}

The Q-table size depends on the number of discrete states and actions. Let the pan
(azimuth) and tilt (elevation) angles be discretized into $N_\theta$ and $N_\phi$
positions respectively, let the RSSI trend take $N_r$ discrete values, and let there
be $N_a$ possible actions. Then, the total Q-table size is:
\begin{equation}
\left|Q\right| = N_\theta \times N_\phi \times N_r \times N_a
\end{equation}

For the deployed controller:  
\begin{itemize}
    \item Pan states: $N_\theta = 19$
    \item Tilt states: $N_\phi = 11$
    \item RSSI trend states: $N_r = 3$
    \item Actions: $N_a = 5$ (left, right, up, down, hold)
\end{itemize}

Hence, the deployed table contains:
\begin{equation}
\left|Q\right| = 19 \times 11 \times 3 \times 5 = 3,135 \text{ Q-values}
\end{equation}
Assuming each Q-value is stored as a 32-bit floating-point number (4 bytes), the
memory requirement is:
\begin{equation}
3,135 \times 4~\text{bytes} \approx 12.3~\text{kB}
\end{equation}
This memory footprint is comfortably within the RAM capacity available on the
ESP32-S3, confirming the feasibility of direct on-device inference.

\subsubsection{Action Space Design}
The action space defines the set of possible antenna movements available to the RL
agent. Actions are discrete and correspond directly to the implemented actuator
commands:

\begin{itemize}
    \item \textbf{Pan left}: decrease azimuth by one $10^{\circ}$ step
    \item \textbf{Pan right}: increase azimuth by one $10^{\circ}$ step
    \item \textbf{Tilt up}: increase tilt by one $5^{\circ}$ step
    \item \textbf{Tilt down}: decrease tilt by one $5^{\circ}$ step
    \item \textbf{Hold}: retain the current orientation
\end{itemize}
Pan actions are executed through the stepper motor driver, while tilt actions are
executed through the servo output. This discrete action set keeps inference simple and
ensures that the learned policy maps directly to repeatable physical motions.

\subsubsection{Reward Function}
The reward function provides evaluative feedback to the reinforcement learning agent by
quantifying the impact of each antenna movement on link quality. The objective is to
increase RSSI while discouraging unnecessary movement that may lead to oscillatory
behavior, increased wear, or slow convergence.

The instantaneous reward at time step $t$ is defined as:
\begin{equation}
r_t = \Delta RSSI_t - \lambda \cdot \mathbb{I}_{\text{move}}
\end{equation}
where:
\begin{itemize}
    \item $\Delta RSSI_t$ is the change in averaged RSSI after executing the action
    \item $\mathbb{I}_{\text{move}}$ is equal to $1$ when a physical movement occurs
    and $0$ for the hold action
    \item $\lambda$ is a weighting factor that penalizes unnecessary actuation
\end{itemize}

A positive reward is obtained when an action results in an improvement in RSSI, encouraging movements that align the antenna more closely with the transmitter. Conversely, actions that degrade RSSI yield negative rewards, guiding the agent away from suboptimal orientations. The motion penalty term discourages excessive or unnecessary antenna movements, promoting stable convergence and reducing oscillations around local maxima.

The parameter $\lambda$ is selected empirically during simulation so that the reward
remains dominated by RSSI improvement while still penalizing inefficient actuation.
This formulation enables the agent to learn a policy that balances communication
performance with mechanical efficiency under embedded constraints.
\subsubsection{Learning and Action Selection}

The reinforcement learning agent operates on a pre-trained Q-table stored on the MCU.
The Q-table is generated offline in the simulation environment using repeated episodes
with randomized starting orientations. During deployment, no further Q-value updates
are performed; the system executes inference only.

\textbf{Action Selection:} 
The agent selects actions using the learned Q-values:
\begin{equation}
a_t = \arg\max_{a} Q(s_t, a)
\end{equation}
where $s_t$ is the current state composed of orientation indices and recent RSSI trend.

No exploration is applied during deployment so that hardware trials remain stable,
repeatable, and comparable across runs.

\subsubsection{Implementation Considerations on ESP32-S3}

\begin{itemize}
    \item \textbf{Memory footprint:} The deployed Q-table occupies approximately
    $12.3$~kB, which is small enough for direct static allocation
    \item \textbf{Computation:} Action selection requires only table lookup and
    simple arithmetic, enabling deterministic runtime
    \item \textbf{Real-time operation:} The ESP32-S3 continuously measures RSSI,
    encodes the current state, retrieves the best action, and commands the actuators
    \item \textbf{Stability:} Since no online learning occurs, the deployed system
    avoids trial-to-trial drift caused by exploration and noisy on-device updates
\end{itemize}

This design ensures that the antenna alignment system is fully autonomous, responsive to changing signal conditions, and feasible for embedded deployment without the overhead of online training.
